# UCS512C RS-485 Protocol — Reverse Engineering Notes

This document describes the proprietary unlock and address-programming protocol
used by UCS512C-based RS-485 RGBW LED fixtures.

All values were obtained by capturing signals with a logic analyzer while
operating a genuine fixture programmer, then iterating with an oscilloscope
until the firmware reproduced the correct waveforms.

---

## Note on PI/PO vs. RS-485 programming

The UCS512C datasheet defines official **PI/PO coding lines** for address
programming — a daisy-chain scheme where each chip passes an incremented
address token to the next. This is the manufacturer-intended method.

Lumino Bridge does **not** use PI/PO. Those lines are only accessible on the
bare chip; finished fixtures do not expose them. The protocol described here
programs addresses entirely over the RS-485 bus, reverse-engineered from
a third-party programmer. This makes Lumino Bridge practical for use with
complete, assembled fixtures where PI/PO lines are not accessible.

---

## Physical layer

- **Interface:** RS-485 half-duplex
- **Baud rate:** 250 000 bps (standard DMX512 rate)
- **Frame format:** 8N2 (8 data bits, no parity, 2 stop bits)
- **Voltage levels:** standard RS-485 differential

---

## Normal DMX frame (used during programming)

```
 LINE HIGH  │  BREAK (LOW)  │ MAB (HIGH) │  Start + Data bytes  │  Gap (LOW)
 T_pre µs   │  via UART 0x00│  T_MAB µs  │  UART write + flush  │  T_inter µs
```

| Parameter | Value | Notes |
|-----------|-------|-------|
| `T_BREAK_PRE_00` | 68 µs | HIGH before BREAK when data is 0x00-type |
| `T_MAB` | 149 µs | Mark After Break |
| `T_INTER_FRAME` | 174 µs | LOW gap between consecutive frames |
| `T_BREAK_PRE_FF` | 155 µs | HIGH before BREAK when data is 0xFF-type |
| `T_POST_FF_1` | 12 µs | Flush settle after 0xFF BREAK byte |
| `T_POST_FF_2` | 12 µs | Flush settle after 0xFF data payload |

> The BREAK is generated by pulling DE/RE LOW and transmitting a `0x00` byte
> at 250 kbaud. The byte itself is ignored by receivers — the idle-LOW period
> (≥88 µs at this baud) is detected as the BREAK condition.

---

## Unlock sequence

The unlock sequence must be sent **once** before address programming.
It consists of two identical "stubs", each made of:

1. A **transition block** — 4 DMX frames with all channels at `0x00`
2. A **0xFF stub** — 1 DMX frame with all channels at `0xFF`, using the
   `T_BREAK_PRE_FF` timing

After each stub, the bus is held **LOW** for a defined period:

| Pause | Duration |
|-------|----------|
| After stub 1 | 2 000 ms |
| After stub 2 | 1 500 ms |

After both stubs, **closing frames** are sent (see below), then the fixtures
are ready to accept address programming.

### Sequence overview

```
[Transition block 1] → [0xFF stub 1] → LOW 2 s
[Transition block 2] → [0xFF stub 2] → LOW 1.5 s
[Closing frames]
```

---

## Closing / init frames

6 frames sent both at the end of the unlock sequence and again at the start
of address assignment ("init frames").

Pattern:

| Frame index | Fixture 1 (pos 0–3) | Fixtures 2–512 (pos 4–2047) |
|-------------|--------------------|-----------------------------|
| 1, 2, 3     | `{0, 0, 0, 0}`     | `{0, 0, 0, 0}` |
| 4, 5, 6     | `{127, 127, 127, 127}` | `{0, 0, 0, 127}` |

The W=127 value on all fixtures in frames 4–6 appears to act as a "handshake"
that signals the fixtures to enter address-receive mode.

---

## Address assignment

After unlock + init frames, addresses are programmed one fixture at a time.

For each address `N` (1 to 512):

1. Build a DMX frame where **only** channels `(N-1)*4` to `(N-1)*4+3` are
   set to `0x7F` (127); all other channels are `0x00`
2. Send the frame **twice** with a 2 ms gap between them
3. Wait `PAUSE_BETWEEN` ms (20 ms) before moving to the next address

The fixture at position `N` sees its slot light up with the marker value and
permanently stores `N` as its DMX start address.

---

## After programming

Fixtures respond to standard DMX512. No special controller needed — WLED,
QLC+, or any DMX source works. WLED with a MAX485 on GPIO 2 tested and
confirmed working.

## Fixture channel map (UCS512C, 4-channel RGBW mode)

| Offset | Channel |
|--------|---------|
| 0 | Red |
| 1 | Green |
| 2 | Blue |
| 3 | White |

## Timing tolerances

The UCS512C accepts a reasonable range around the values above. During
development, values within ±20% of the targets were sufficient for reliable
programming. The values in this document are what the genuine programmer
produces; staying close to them is recommended for maximum compatibility.
